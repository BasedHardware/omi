# T-020 — Pass sender + recent messages to persona-chat

## Context

The T-019 prompt rewrite fixed the "AI clone" framing leak, but the bot still doesn't know two things:

1. **Who it's talking to.** A message from "Alice" looks identical to a message from "Bob" — the persona has no sender name, no platform hint, no relationship context. So "who am I?" / "remind me about X" / "are you a bot?" all get generic answers.
2. **What was said 30 seconds ago.** Telegram sends each message as a fresh webhook; the LLM has zero memory between messages. "yo / what's up / I'm looking for a coffee shop in Asok" loses the thread after the second message.

T-020 wires both. Backend schema accepts `context` + `previous_messages`; the route injects them into the prompt; both plugins populate them from their respective message payloads and a per-chat ring buffer.

## Wire formats

### Request body
```json
{
  "text": "yo",
  "context": {
    "sender_name": "Alice",
    "sender_username": "alice_t",
    "chat_type": "private",
    "platform": "telegram"
  },
  "previous_messages": [
    {"role": "human", "text": "hi"},
    {"role": "ai", "text": "hey, what's up"},
    {"role": "human", "text": "what's my favorite coffee?"},
    {"role": "ai", "text": "you love pour-over, dark roast"}
  ]
}
```

All fields optional. Backward-compatible with existing callers.

### Backend message assembly
The current persona route builds:
```
[SystemMessage(persona_prompt), HumanMessage(text)]
```
T-020 makes it:
```
[SystemMessage(persona_prompt), SystemMessage(context_block or ""), HumanMessage(prev0), AIMessage(prev0_reply), ..., HumanMessage(current_text)]
```
`context_block` is rendered only if `body.context` is present:
```
You are talking to {sender_name} on {platform}. This is a {chat_type} chat.
```
The persona handler in `execute_persona_chat_stream` already sets the persona_prompt SystemMessage, so we add the context SystemMessage AFTER it in the route (insert at index 1).

### Per-chat ring buffer
- Stored on the user dict in `simple_storage` as `recent_messages: list[{role, text, ts}]`
- Max 10 entries (5 turns). Older entries drop off FIFO.
- Appended after `_persona_chat` returns successfully (so failed replies don't poison history).
- Bot's own "/start" handshake reply is NOT appended (it's not part of the persona conversation).

## Risks

- **Backward compat:** All new fields are Optional. Existing tests calling the route without `context`/`previous_messages` continue to pass.
- **PII in logs:** The `context` dict will contain sender names. Backend logs already sanitize response.text, but the request body isn't auto-sanitized. Keep context rendering in `_stream()` and don't echo the body in error logs.
- **Memory growth:** Ring buffer is bounded to 10 entries per chat. With 1000 chats the on-disk JSON is ~1MB — fine.
- **Concurrency:** `simple_storage` writes the whole file on every append. If a chat fires 10 messages/sec, we'd rewrite the file 10x/sec. Per-chat lock (or moving to Redis) is a future concern; for v0.1 it's fine since the file is small.

## Out of scope

- iMessage plugin (T-009 follow-up) — same pattern applies but no plugin exists yet.
- Redis-backed storage — current file-based storage is fine for v0.1.
- TTL on ring buffer entries — 10-entry FIFO is enough for short conversations; older context drops naturally.
