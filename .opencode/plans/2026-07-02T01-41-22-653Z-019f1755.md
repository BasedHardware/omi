# Plan: "Reply as Yourself" via telegram-mcp (replace plugins/omi-telegram-app)

## Summary

Replace the **Bot-API** model of `plugins/omi-telegram-app` (separate bot account, webhook-based, replies appear from a bot identity) with a **Telethon/User-API** model in which the AI Clone reply is sent from the user's personal Telegram account via `chigwell/telegram-mcp`. Receipt switches from webhooks → long-poll.

This is a **subset** of the AI Clone: only the Telegram plugin is rewritten. WhatsApp (already a verified-business API) and the desktop/backend persona layer stay.

---

## 1. Why a separate branch / PR, not appended to PR #8682

PR #8682 is *already* large (122 commits, 30+ files, three prompt-injection review rounds, a recent rebase). Adding an entirely new auth model + transport on top would:
- Force a fourth security review round on the existing PR
- Make the existing prompt-injection fixes harder to bisect
- Couple the ToS/banning risk of user-account automation with the Bot API change

The right place: **new branch `feat/ai-clone-user-account-telegram`**, opens a fresh PR. Reuses everything from #8682 that doesn't depend on the bot-token model (persona prompt template, memory RAG, sender context framing, recent-messages ring buffer, AICloneConfig discovery, etc.).

## 3. Auth model — what's actually different

| Concern | Old (Bot API) | New (User API / Telethon) |
|---|---|---|
| **Credential** | `bot_token` (revocable from BotFather) | `api_id` + `api_hash` + `TELEGRAM_SESSION_STRING` |
| **Where it comes from** | User pastes from @BotFather | User runs one-time `session_string_generator.py` against their **phone number** |
| **Revoke-ability** | One-click revoke in BotFather | Must delete session AND revoke session in Telegram Settings → Devices (every device) |
| **Identity on the wire** | bot account | the user's personal account (literally `from_user == you`) |
| **Multi-device** | Bot lives on every desktop that runs the plugin | Each desktop that runs MCP shows up as another device in the user's Telegram UI |
| **Auth scope** | `/setup` then webhooks | long-poll on Telethon socket — no per-chat handshake |
| **First-run friction** | low — copy/paste | high — must scan QR or enter SMS code |
| **ToS posture** | approved path | userbot gray zone; Telegram can ban the account |

The session string is **the user's phone-number authentication**. Anywhere it lands, treat it as more sensitive than any other credential in the system. A leaked session string = a fully compromised personal Telegram identity, not a leaked bot that can be revoked.

## 4. New plugin: `plugins/telegram-user-account/`

Mirrors the shape of `plugins/omi-telegram-app/` minus the bot handshake + webhook. New service exposing:

| Method | Path | Purpose | Auth |
|---|---|---|---|
| `GET` | `/health` | liveness | none |
| `GET` | `/status` | `{connected: bool, account_phone: string, account_name: string, unread_chats: int}` — what the user sees in the desktop banner | bearer |
| `GET` | `/recent_messages` | list of recent personal-account chats with `{chat_id, title, last_message_preview, unread_count, auto_reply_enabled}` so the desktop can render a chat list | bearer |
| `GET` | `/recent_messages/{chat_id}/messages?limit=20` | per-chat ring buffer (replaces `/setup` + webhook history) | bearer |
| `POST` | `/persona_chat` | payload `{chat_id, text, sender_handle?}` → persona API → reply string → sent via `mcp.send_message(chat_id, reply)` — no webhook feedback path needed | bearer (omi_dev key + per-user check) |
| `POST` | `/chat_memory` | payload `{chat_id, role, text}` → append to per-chat in-memory ring buffer (replaces webhook-side persist) | bearer |

### How it's wired internally

```
FastAPI service (port 18800)
        │
        ├─ subprocess: telegram-mcp (Telethon session)
        │       │  exposes: send_message, get_messages, get_chats,
        │       │             search, react, ...  (80+ tools)
        │       ▼
        │  Telethon → MTProto → telegram cloud (as the user)
        │
        └─ outbound HTTPS → backend /v2/integrations/.../persona-chat
```

- The plugin service runs `telegram-mcp` as a child process via `asyncio.create_subprocess_exec` and speaks to it via `MCP stdio transport` (the standard MCP wire format). Process supervision: restart on exit, log PID, exponential backoff.
- Outbound persona calls use the same `plugins/_shared/persona_client.chat()` that the bot plugin uses today. Auth is still the per-user `omi_dev_*` key.

### Storage

A new `simple_storage.py` (mirrors the chat-history pattern of `_shared/storage`):

| File | Contents |
|---|---|
| `users_data.json` | `chat_id → {auto_reply_enabled, recent_messages: [{role, text, ts}], last_seen_at, omi_uid, persona_id}` |
| `account.json` | `{phone: string, name: string, device_label: string}` — populated from Telegram's `get_me()` at startup |

`api_id` / `api_hash` / `TELEGRAM_SESSION_STRING` are **never written to disk**. They live in env vars passed to the subprocess, or in macOS Keychain (desktop side).

### Webhook replacement — incoming messages

The old plugin was *pushed* updates via webhook. The new plugin is *pulled*:

- MCP `events` tool emits new messages asynchronously (Telethon `NewMessage` event handler).
- Plugin service queues them and persists `{role: 'human', text, ts}` to the chat's ring buffer.
- On `/recent_messages/{chat_id}/messages` the plugin returns the buffer to the plugin service which (in turn) feeds it to `/persona_chat` as `previous_messages`.

### No handshake

There's no `/setup` ↔ `/start` handshake. The plugin is "connected" the moment the Telethon session starts polling and `get_me()` succeeds. The desktop UI replaces the "waiting for /start" step with **"your Telegram account: choguun (+66...)"** showing the logged-in identity, which is the new "Connected" signal.

## 5. Desktop side: `AICloneConfig` schema additions

```swift
// AICloneConfig.swift — new published fields

/// Personal Telegram account for the User-API integration.
/// nil = legacy Bot-API plugin still in use; non-nil = user-account mode is active.
/// Switching between them is a per-AICloneConfig setting, not a global flag.
struct TelegramAccountConfig {
    let apiId: Int
    let apiHash: String       // not secret — public on my.telegram.org anyway
    let sessionLabel: String  // "work" / "personal" / custom; used in MCP `account` parameter
}

@Published var telegramAccount: TelegramAccountConfig? = nil

/// Keychain item: encrypted Telethon session string. Created via the
/// session_string_generator.py flow + keychain Migrate-at-setup. NEVER
/// leaves this process; never logged; never sent over the wire to the
/// plugin service (the session string is passed to the MCP subprocess
/// via a per-pid pipe, never an HTTP request).
```

### `Keychain.swift` adds a new item

```swift
case telegramSessionString(label: String)  // value is session string
```

Multiple sessions can live in Keychain keyed by their `sessionLabel`, e.g. one per workspace.

### `PluginDiscovery.swift` adds a new file path

Old: `~/.config/omi/ai-clone-plugin.json` (bot plugin)
New (alongside, NOT replacing): `~/.config/omi/ai-clone-telegram-user.json`

```json
{
  "version": 1,
  "plugin_url": "http://127.0.0.1:18800",
  "bearer_token": "local-dev-token-...",
  "account_phone": "+66xxxxxxxxx",
  "account_name": "choguun",
  "device_label": "Omi Desktop (MacBook Pro 16,2)",
  "omi_base_url": "https://api.omi.me",
  "started_at": 1782833712.13,
  "instance_id": "..."
}
```

`account_phone` is **metadata only** — never used for auth. Auth still happens via `TELEGRAM_SESSION_STRING` carried in the subprocess's environment, not via HTTP.

### Connect flow rewrite

`ConnectSheet.swift` is the user-facing surface. New fields for Telegram:

```
Telegram setup form (when plugin == .telegram && useUserAccount = true):
  - Account label: "Personal" / "Work" / custom
  - api_id (number)
  - api_hash (text)
  - Session string (text, secure)
  - "Generate session" button → spawns `python session_string_generator.py` in a subprocess,
    captures stdout, pipes the session string back into the form (one-shot)
  - Status: "Logged in as: choguun (+66xxxxxxxxx) — Choguun_Omi_TG_bot" → etc.
```

The QR / deep-link step goes away. There's no `/start` to send.

### Discovery-bridge: still needed but different

Today the desktop reads `ai-clone-plugin.json`. With TWO plugins (bot + user-account) both writing discovery files, the desktop needs to:
1. Read both
2. Prefer the user-account one if present (more recent / set as preferred)
3. Fall back to the bot one

Decision: keep two discovery files. Add `preferUserAccount: Bool` to AICloneConfig (defaults to false; opt-in via a new toggle in Settings → AI Clone → Telegram section: "Reply as me (personal account)"). When the user toggles it on, the desktop instructs `telegram-user-account` to take over.

## 6. Backend side: zero changes needed for the swap

`backend/routers/integration.py` already takes `context` (sender_name, sender_username) and `previous_messages` (chat history). The user-account plugin populates `sender_name` from the personal-account user's display name (Telegram allows users to set first_name freely — the same prompt-injection framing from #8682 round 7 applies unchanged).

The `verify_api_key_for_uid` check still happens. Per-`omi_uid` key pinning: still required, still in `verify_api_key_for_uid`.

The `context` sender_name field becomes **even more sensitive** in user-account mode — the sender is the user themselves, but the prompt still gets to see the Telegram profile first name (which a user can set to anything). No backend change; same `sanitize_context_field` from round 7.

## 8. ToS / ban-risk mitigation (build it in, don't bolt it on)

The maintainer flagged long-lived-credential handling repeatedly. The session string is the *most* such credential. Default posture:

| Mechanism | Status |
|---|---|
| Read-only mode (`TELEGRAM_EXPOSED_TOOLS=read-only`) supported by MCP | **default ON** during setup; user toggles write tools on after first confirmed send |
| Per-message desktop UI confirmation | for the first 50 outbound messages: desktop shows the draft reply in a "Send?" toast; user must approve each one. After 50 it learns and goes silent. Toggle in Settings. |
| Rate-limit cap (e.g. ≤ 30 outbound messages/hour/chats) | plugin-side cap matching Telegram's anti-flood numbers |
| Clear "this is your personal account" banner in the connect sheet | yes, with a one-time acknowledgement checkbox |
| Ban-warning panel in plugin log when Telegram returns `FLOOD_WAIT_*` errors | yes, surfaces in desktop log + warning status in PluginURLCard |
| Daily "messages sent today" counter on the plugin card | yes (cosmetic, but makes flooding visible to the user) |

## 10. Open questions to flag for the maintainer review

1. **Is this in scope for the AI Clone track?** PR #8682 already optimizes the Bot API path. This is a *second* AI Clone mode. The contest rubric scores "reply as you" as a product differentiator, but the ToS risk may not be worth the marginal UX.
2. **Keychain vs. encrypted file for the session string?** macOS sandboxing could conceivably cause issues. Recommend starting with Keychain, fall back to encrypted file only if Keychain access fails (which shouldn't happen in our signing context).
3. **Trust UX for the first 50 messages** — too annoying? ship with just the rate-limit + flood-detect, no confirm-per-message? Probably yes for v1; can add the per-message confirm later if needed.
4. **Multi-account** — do we expect users to want their work persona and personal persona both backed by MCP? Ship with one session per `AICloneConfig` first; add the multi-account picker later.
5. **Codemagic signing identity** — does the desktop bundle already have the right keychain group/team ID, or does this need an entitlements update?
