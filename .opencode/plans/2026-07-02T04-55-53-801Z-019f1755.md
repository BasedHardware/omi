# Plan: Auto-reply toggle for AI Clone (Mac desktop, Swift)

## 1. Problem statement

The Mac desktop (Swift) has an auto-reply toggle for the **bot**
AI Clone mode (PluginCard.swift calls `POST /toggle` on the
`omi-telegram-app` plugin). It does NOT have one for the
**user-account** mode (Telethon, added in PR #8822).

Worse, the user-account plugin's HTTP API has no `/toggle`
endpoint at all, and `/persona_chat` does not check
`auto_reply_enabled` before calling the persona API. The
`auto_reply_enabled` field exists in `simple_storage.py`
(via `update_auto_reply(telegram_user_id, enabled)`) but
nothing reads it on the request path. Result: turning off
auto-reply requires signing out entirely, and the desktop
has no UI for it.

## 3. Design

### 3.1 Plugin-side (Python, `plugins/telegram-user-account/`)

**3.1.1 Add `POST /toggle` endpoint** mirroring the bot
plugin's API shape but using `handle` instead of `chat_id`
(because the user-account flow keys storage by Telegram
user handle, not chat id):

```python
class ToggleRequest(BaseModel):
    handle: str = "all"     # "all" toggles all users (the typical
                            # case; the user-account flow is
                            # single-account so "all" = "the one user")
    enabled: bool

class ToggleResponse(BaseModel):
    auto_reply_enabled: bool
    affected_users: int

@app.post("/toggle", response_model=ToggleResponse, dependencies=[Depends(require_bearer)])
async def toggle(req: ToggleRequest):
    if req.handle == "all":
        affected = 0
        for telegram_user_id in list(simple_storage.users.keys()):
            simple_storage.update_auto_reply(telegram_user_id, req.enabled)
            affected += 1
        if affected == 0:
            raise HTTPException(status_code=403, detail="No users configured")
        return ToggleResponse(auto_reply_enabled=req.enabled, affected_users=affected)
    if req.handle not in simple_storage.users:
        raise HTTPException(status_code=403, detail="Unknown handle")
    simple_storage.update_auto_reply(req.handle, req.enabled)
    return ToggleResponse(auto_reply_enabled=req.enabled, affected_users=1)
```

**3.1.2 Add `auto_reply_enabled` to `/status`** so the
desktop's existing 30s poll can keep the toggle in sync:

```python
# In /status handler
auto_reply_states = [
    u.get("auto_reply_enabled", False)
    for u in simple_storage.users.values()
]
return {
    "connected": ...,
    "auto_reply_enabled": any(auto_reply_states) if auto_reply_states else False,
    "rate_limit": rl_state,
    ...
}
```

(Returns `True` if any user has auto-reply on ‚Äî the desktop
treats it as a global toggle, so any-true == on.)

**3.1.3 Gate `/persona_chat` on `auto_reply_enabled`**.
Add a check at the top of `persona_chat_endpoint`:

```python
if not user.get("auto_reply_enabled", False):
    logger.info("auto_reply disabled for %s; skipping persona call",
                body.sender_handle)
    raise HTTPException(
        status_code=403,
        detail="Auto-reply is disabled. Enable it in Omi desktop."
    )
```

**3.1.4 Update stale docstring** in `/status` (still
references the "per-chat ring buffer" wording from
cubic review 4618627789 P2; the implementation was
updated but the docstring was missed).

**3.2.3 Add `AICloneClient.toggleUserAccount(enabled:)`**
method:

```swift
func toggleUserAccount(
    baseURL: String,
    bearerToken: String,
    enabled: Bool
) async throws {
    let url = try endpointURL(baseURL: baseURL, path: "/toggle")
    var request = URLRequest(url: url)
    request.httpMethod = "POST"
    request.setValue("Bearer \(bearerToken)", forHTTPHeaderField: "Authorization")
    request.setValue("application/json", forHTTPHeaderField: "Content-Type")
    let body: [String: Any] = ["handle": "all", "enabled": enabled]
    request.httpBody = try JSONSerialization.data(withJSONObject: body)
    let (data, response) = try await session.data(for: request)
    guard let http = response as? HTTPURLResponse, (200...299).contains(http.statusCode) else {
        let code = (response as? HTTPURLResponse)?.statusCode ?? -1
        throw AICloneError.network("Plugin returned HTTP \(code)")
    }
    // Decode response to verify
    struct Response: Decodable {
        let auto_reply_enabled: Bool
        let affected_users: Int
    }
    _ = try JSONDecoder().decode(Response.self, from: data)
}
```

**3.2.4 Add `autoReplyEnabled: Bool?` to `StatusResponse`**
for the user-account schema, alongside the existing
`rateLimit` and `messagesSentToday` fields. Already
optional (matches the dual-schema decode pattern).

**3.2.5 Wire the toggle in `ConnectSheet`** to call
`toggleUserAccount`, with rollback on error:

```swift
private func flipTelegramAutoReply(enabled: Bool) async {
    telegramAutoReplyInFlight = true
    defer { telegramAutoReplyInFlight = false }
    do {
        try await AICloneClient.shared.toggleUserAccount(
            baseURL: config.pluginURL,
            bearerToken: config.bearerToken,
            enabled: enabled
        )
    } catch {
        // Roll back the UI state
        await MainActor.run { config.telegramAutoReplyEnabled = !enabled }
    }
}
```

### 3.3 Why not a separate `UserAccountCard`?

A separate card on `AIClonePage` (sibling to the existing
Telegram/WhatsApp cards) was considered. Rejected because:

- The "Reply as me" UX already lives in the ConnectSheet
  (per plan ¬ß7) ‚Äî adding a second card would split the
  connected-state experience across two places.
- `AICloneConfig` already has `telegramAccountEnabled` and
  the 30s `/status` poll. Reusing it is less new code.
- The bot-mode card has a different shape (per-chat
  `connected_chats`, `bot_username`). Mixing them into
  one card would conflate two distinct modes.
- The user-account flow is one-account-one-user; the
  "all" semantic in `/toggle` is a natural fit for a
  single toggle in the connected-state section.

## 6. Out of scope (deferred)

- **Per-chat toggle.** The user-account flow keys storage
  by user handle, not chat id. A per-chat toggle would
  require a schema change to the user record (e.g.
  `chat_id -> auto_reply_enabled` map). Not in this PR.
- **ToS confirmation on toggle.** Toggling auto-reply off
  does not need a re-acknowledgement (the user already
  accepted ToS at setup time). Toggling ON is the same
  path; the ToS gate is at the Setup step, not the toggle.
- **WhatsApp user-account mode.** The current user-account
  flow is Telegram-only (Telethon). WhatsApp would need a
  separate implementation. Out of scope.
- **Backend-side enforcement.** The current Omi backend
  trusts the plugin's persona_chat call. If the desktop
  lies about `auto_reply_enabled`, the persona API still
  gets called. Defense in depth would add a check in the
  backend; not in this PR.
- **Telethon session renewal on toggle.** A revoked
  session would fail the /persona_chat with 502, not the
  403 the toggle is meant to produce. Out of scope (the
  "is_blocked" rate-limit surface already covers the
  common case).

## 7. Risk + mitigation

| Risk | Mitigation |
|---|---|
| `/persona_chat` is currently permissive ‚Äî adding the `auto_reply_enabled` gate could break in-flight calls that haven't yet been told about the change | The gate reads from `simple_storage` which is JSON-file backed; default `False` for users with no field means existing setups are gated OFF on first deploy. Document this in the changelog as a one-time "you must re-enable auto-reply" step |
| The `telegramAutoReplyEnabled` is in-memory only ‚Äî plugin restart loses it | Mitigation: the 30s `/status` poll re-syncs on next launch. No durability needed. The plugin's `simple_storage` IS the durable source of truth |
| The "all" toggle semantic hides per-user intent | Mitigation: the `/status` response exposes the global state only. If a future per-user toggle is needed, add a `handles` array to the response |
| The ConnectSheet's `userAccountSection` grows crowded with the new toggle + Sign out + rate-limit badge | Mitigation: layout test (visual, not automated) ‚Äî the connected-state already had 4 widgets, adding the toggle is 1 more. If the sheet is too crowded, future PR can extract a `userAccountCard` |

## 10. Open questions for the maintainer

1. **Default value on first deploy.** The new
   `auto_reply_enabled` gate defaults to `False` for any
   user record that doesn't have the field. This is the
   safe default (no auto-reply until the user opts in),
   but it means existing users will need to re-enable
   auto-reply after this PR lands. Is that acceptable, or
   should we backfill `auto_reply_enabled: true` for
   existing user records in a one-time migration?

2. **Per-user vs global toggle.** The plan uses
   `handle: "all"` for the global toggle and reserves
   per-handle for future use. Is "all" the right default
   semantic, or should the toggle be per-handle from the
   start? (Per-handle requires the desktop to know the
   current user's handle, which it does have from the
   discovery file's `phone` field.)

3. **Toggle position in the connect sheet.** Adding the
   Auto-reply toggle to the `userAccountSection` connected
   state places it next to "Sign out". Is that the right
   visual grouping, or should it be more prominent (e.g.
   the first row in the connected state)?
