# R4 Plan — Nightly Rotation Cron (updated per David 2026-07-02)

## Origin (David's feedback)

> - MVP promotion path should be eval → promotion PR → human review / merge
> - Cron / automation can propose promotion PRs, but shouldn't directly mutate serving routes yet, once gateway is stable and we're comfortable with the promotion path we can then go to full auto
> - External benchmarks alone are not enough for promotion, we will need to have our own internal evaluation set for the stuff that we care about, at least as a gate (table for later)
> - Fake smoke is fine as a unit harness, but the actual gate should hit the real gateway path

## Promotion path (R4)

### Flow (per the MVP)

```
Catalog (lanes_catalog.yaml)
   ↓
Internal eval set (gates promotion) — separate work, "table for later"
   ↓
Promotion PR (proposed by cron, reviewed + merged by humans)
   ↓
Serving config (lanes.yaml + route_artifacts.yaml)
   ↓
R5b hot-reload picks up the change
```

### R4's role in the flow

- **R4 is the cron**: it scans the catalog, finds `dev_only` lanes with completed internal eval, generates a promotion PR
- **R4 does NOT directly mutate the serving config**: it generates a PR; humans review + merge
- **R4 does NOT auto-merge**: the PR is the decision (per PLAN.md)
- **Eventually**: once the promotion path is stable + the team is comfortable, R4 can auto-merge. Not now.

### Internal eval gate (separate work)

Per David: "External benchmarks alone are not enough for promotion, we will need to have our own internal evaluation set for the stuff that we care about, at least as a gate (table for later)."

This is a separate work item. R4's design assumes the internal eval exists and is referenced from the catalog's `eval_suite` field. R0.5's catalog already has `eval_suite: null` for `dev_only` lanes — this is the design hook for the internal eval.

When the internal eval exists, the promotion PR generated by R4 would include:
- The new lane entry in `lanes.yaml`
- The new artifact in `route_artifacts.yaml`
- A link to the internal eval results (proving the lane passed the gate)

The eval is the gate. Cron proposes. Humans merge.

### Real smoke as the actual gate

Per David: "Fake smoke is fine as a unit harness, but the actual gate should hit the real gateway path."

R2's smoke is the unit harness. R4's cron uses a REAL smoke — calls the live gateway's HTTP endpoint, not a fake. This is what the gate actually checks.

The cron workflow:
1. Run the emitter with `--dry-run` (R1's emitter, plan-only output)
2. Run the real smoke against the gateway's HTTP endpoint (R2's smoke, real provider)
3. Check the internal eval results for each candidate lane
4. If all three pass, generate a promotion PR

### R4 workflow file (planned, not implemented yet)

`.github/workflows/llm-gateway-nightly-rotation.yml`:
- Schedule: daily, off-minute
- Steps:
  1. Checkout + setup Python
  2. Run R1's emitter (dry-run)
  3. Run R2's smoke (real provider, against the live gateway)
  4. Check the internal eval results for each candidate lane
  5. Generate a promotion PR (with a structured body — see below)
- `concurrency: group: nightly-rotation, cancel-in-progress: true`
- No auto-merge
- Required reviewers: gateway owners team
- Required check: R2's smoke + the internal eval

### Promotion PR body (planned format)

```markdown
# Promote <lane-id> to serving config

## Lane
- id: omi:auto:chat-extraction
- provider: openai
- model: gpt-4.1-mini
- surface: openai.chat_completions

## Internal eval results
- suite: backend/llm_gateway/eval/chat_extraction.v1.json
- pass_rate: 0.94
- threshold: 0.90
- ✅ pass

## Real smoke results
- lane: omi:auto:chat-extraction
- capability smoke: pass (text_input ✓, structured_output ✓)
- latency_p95_ms: 230
- timeout_ms: 8000
- ✅ pass

## Diff
- lanes.yaml: +1 lane
- route_artifacts.yaml: +1 artifact (active + LKG)
- digest(s): sha256:...

## Reviewer checklist
- [ ] Internal eval results reviewed
- [ ] Smoke results reviewed
- [ ] Latency acceptable
- [ ] No regression in /ready endpoint
```

The cron generates this PR automatically. Humans review and merge. After merge, R5b's hot-reload picks up the new lane + artifact.

## When to go full auto (R4.1+)

Once the promotion path is stable + the team is comfortable, R4 can be extended to auto-merge. The preconditions:
- ≥30 days of human-in-the-loop merges with no rollback
- Internal eval is rock-solid (catches regressions)
- The cron has been observed doing the right thing

Not now. R4 ships with human-in-the-loop.

## Implementation (planned, not yet started)

R4's implementation is a separate cycle (after R3.2). The flow:
1. `.github/workflows/llm-gateway-nightly-rotation.yml` — the cron
2. The promotion PR generator — reads the catalog, runs the eval, runs the smoke, generates the PR body
3. Tests: workflow fixture with a fake GitHub API + a fake internal eval
4. Run-mode integration test: end-to-end on a real workflow run (in CI)

## Acceptance criteria (planned, after implementation)

1. The cron runs daily and generates a promotion PR per candidate lane
2. The promotion PR body has all required fields (lane, eval results, smoke results, diff, reviewer checklist)
3. No auto-merge
4. Internal eval is the gate (R4 rejects candidates that don't pass)
5. Real smoke is the secondary gate (R4 rejects candidates that don't pass)
6. The promotion PR is parseable by humans (markdown, no jargon)

## Out of scope (planned)

- Internal eval set design — separate work, "table for later"
- Auto-merge — R4.1+ future
- Real provider for R2's `--provider real` (currently R3.2 scope)
