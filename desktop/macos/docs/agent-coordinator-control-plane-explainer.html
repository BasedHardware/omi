What the new control plane does

This PR turns scattered desktop agent surfaces into a local coordinator control plane. The TypeScript runtime and omi-agentd.sqlite3 are now the durable authority; Swift, voice, task chat, Ask Omi, and automation read from and call into that authority.

Old Setup vs New Setup

The old path was surface-led. The new path is coordinator-led and auditable.

Before: surface-led agent behavior

Each surface knew just enough to start or display agent work, but there was no single local coordinator view.

Ask Omi
Prompt text could carry recent chat context directly into worker requests.
Task chat
Task-specific context was prepended locally instead of persisted as an explicit context packet.
Voice
Status summaries were tied to legacy task-agent/floating-pill projections.
Approvals
Sensitive requests had policy metadata, but approval/grant resolution was not a complete coordinator loop.
UI state
Swift presentation paths could drift toward product truth unless carefully constrained.

After: coordinator-led control plane

Surfaces call control tools; the runtime persists the durable facts and derives the UI-facing views.

Ask Omi
Builds a DesktopContextPacket before calling the worker path.
Task chat
Uses task-scoped context packets with provenance, TTL, retention class, and redacted previews.
Voice
Reads coordinator open loops through get_desktop_open_loops.
Approvals
Creates durable dispatches; trusted user control resolves dispatches and may mint scoped grants.
UI state
Swift calls the runtime and displays projections; SQLite remains source of truth.

Request Flow

A user request now passes through explicit routing, context scoping, policy, execution, and review loops.

1

Surface receives intent

Main chat, task chat, voice, or automation starts with user intent plus a surface kind.

2

Coordinator routes

route_desktop_intent checks open loops and reusable session candidates before starting new work.

3

Context packet is persisted

build_desktop_context_packet stores selected snippets, provenance, preview, TTL, and access rows.

4

Policy gates sensitive work

Screenshots, writes, task mutation, and broad context require explicit approval dispatches.

5

Surfaces project state

Swift displays action queues and open loops derived from runtime state, not local UI guesses.

Durable State Added

The coordinator extends the existing agent SQLite store. These are not separate databases or UI caches.

desktop_context_packets
  • Scoped context for worker prompts.
  • Stores redacted preview, hash, retention class, TTL.
  • Expired packets are removed during startup reconciliation.
desktop_dispatches
  • Durable user attention and approval items.
  • Kinds include approval, routing choice, failure recovery, artifact review, candidate review, and screen context.
  • Pending expired dispatches become expired with reconciliation metadata.
desktop_artifact_deliveries
  • Tracks whether an agent artifact reached the intended surface.
  • Review status and delivery status are separated.
  • Retrying deliveries are failed on daemon startup reconciliation.
desktop_memory_candidates
  • Agents propose memory facts instead of silently writing them.
  • Includes evidence refs, confidence, sensitivity, and review status.
desktop_task_candidates
  • Agents propose task creates, updates, completes, and deletes.
  • Can require approval before state mutation.
desktop_context_access_log
  • Audits context packet sources and operations.
  • Links sensitive context access to dispatch ids when required.

Approval Boundary

The coordinator now distinguishes model-visible inspection from trusted user-control resolution.

Allowed read path

Inspect and summarize

Agents can ask for awareness snapshots, action queues, open loops, intent routing, and policy evaluation. These do not directly mutate approval state.

build_desktop_awareness_snapshot list_desktop_action_queue get_desktop_open_loops
Dispatch required

Sensitive context

Screen images, screen history, and sensitive snippets require a verified approved dispatch id before they can enter a context packet.

screen_context approval dispatch_id
Not model-visible

Resolve approvals

resolve_desktop_dispatch is restricted to trusted direct control. The pi-mono projection explicitly excludes it from model-advertised tools.

trustedUserControl scoped grants finite expiry

Swift local policy denies risky tools first

ChatToolExecutor.localPolicyDecision denies SQL writes, task mutation tools, and screenshot image access with structured policy-denied messages.

Runtime validates grants before mutation

resolveDesktopDispatch only mints grants for approval dispatches where capability, operation, resource, owner, session, and expiry match.

Surface Integration

Each user-facing surface got a concrete coordinator touchpoint.

Main chat

Builds a main-chat context packet from recent messages and the current user message, with a 15-minute TTL and boundary policy for task/memory/screenshot access.

  input: mainChatPacketInput

Builds task-scoped packets instead of stuffing raw task context into prompt text. The worker sees a packet id and redacted preview.
