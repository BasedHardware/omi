# Mac→Windows Parity Audit — Proactive Assistants: Focus & Insight

> Scope: the screen-context proactive engine — orchestration framework, Focus (distraction detection + glow), and Insight (proactive tips). Windows baseline checked: `src/main/insight/`, `src/main/ipc/insight.ts`, `src/main/ipc/db.ts`, `src/renderer/src/lib/insightEngine.ts`, `insightGate.ts`, `insightPrompt.ts`, `insightActivity.ts`, `insightGate.test.ts`, `insightPrompt.test.ts`, `insightActivity.test.ts`, `src/renderer/src/components/insight/InsightToast.tsx` + `.css`, `src/renderer/src/lib/geminiClient.ts`, `src/renderer/src/lib/screenRedact.ts`, `src/renderer/src/components/settings/tabs/RewindTab.tsx`, `src/renderer/src/App.tsx`, `src/preload/index.ts`, `src/shared/types.ts`. Grepped the whole `src/` tree for `focus`, `distracted`, `glow` and found no Focus-assistant equivalent anywhere.

## Pluggable assistant coordinator

- **What it is:** The framework all proactive features plug into: a shared `ProactiveAssistant` actor protocol (`shouldAnalyze`, `analyze`, `handleResult`, `onAppSwitch`, `onContextSwitch`, `clearPendingWork`, `stop`), a coordinator that fans out captured frames to every registered assistant with per-assistant backpressure (skip if still analyzing), and a set of small pure-function policy gates (`ProactiveAssistantOrchestrationPolicy`) governing when to pause capture (screenshot apps frontmost), throttle capture (video calls), and debounce/flush frame distribution on context change.
- **Where (Mac):** `Core/AssistantCoordinator.swift`, `Core/AssistantProtocol.swift`, `Core/ContextDetection.swift`, `Core/ProactiveAssistantOrchestrationPolicy.swift`, `Core/CapturedFrame.swift`, wired together in `ProactiveAssistantsPlugin.swift`.
- **How it works:** `ProactiveAssistantsPlugin` runs a capture timer (interval driven by `RewindSettings`, power-aware), gates on screen-recording permission, video-call/screenshot-app throttling, and `ContextDetection.didContextChange` (app or normalized window-title change — strips spinners/timers/counts so noisy titles like a Claude Code progress spinner don't cause spurious re-analysis). On a real context change it starts a configurable "analysis delay" (`AssistantSettings.analysisDelay`) before distributing frames, so the model isn't analyzing mid-switch. `AssistantCoordinator.distributeFrame` fans the frame out to every registered assistant (Focus, Task, Insight, Memory) with a `isAnalyzing` set preventing frame pile-up per assistant. All local, no backend involvement in the orchestration itself.
- **Windows status:** Absent as a generalized framework. Windows' equivalent is a single self-rescheduling `setTimeout` loop in `insightEngine.ts` that is Insight-specific — there's no shared protocol, no per-assistant backpressure, no context-switch detection, and no debounce/flush distribution gate. Checked: no `distributeFrame`/`shouldAnalyze`/`onContextSwitch`-shaped abstraction anywhere in `src/main` or `src/renderer/src/lib`.
- **Value / notes:** Medium — matters mainly because it's the scaffold a Windows Focus assistant (see below) would need; not directly user-visible on its own. macOS API note: window-title/app polling here uses `NSWorkspace` + Accessibility; Windows already has an equivalent via `nativeForeground.ts` (used by the usage tracker), so the primitives exist, just not wired into a coordinator.

## Focus Assistant

- **What it is:** A background loop that periodically screenshots the active window, asks Gemini "is the user focused or distracted right now" (with the user's goals/tasks/memories/profile as context), and reacts: shows a colored glow around the window (green = refocused, red = distraction), sends a short coaching notification, records a focus/distracted session, and enters a cooldown so it doesn't re-analyze every frame while already-known-distracted.
- **Where (Mac):** `Assistants/Focus/FocusAssistant.swift` (actor, ~860 lines), `FocusModels.swift` (`FocusStatus`, `ScreenAnalysis`), `FocusStorage.swift` (session history + stats), `FocusAssistantSettings.swift` (prompt, cooldown, exclusions), `UI/FocusTestRunnerWindow.swift` (dev tool).
- **How it works:**
  - Runs off the shared frame pipeline (`analyze(frame:)` skips lock-screen and excluded apps, and — the key optimization — skips re-analysis entirely if the app+window context hasn't changed and the user is currently `.focused`, so a static IDE window doesn't burn Gemini calls).
  - Sends a single-image Gemini request (`GeminiClient.sendRequest(prompt:imageData:...)`, JSON schema `{status, app_or_site, description, message}`) with `refreshContext()` prepending: AI user profile text, current time, up to 10 active goals, up to 50 tasks by priority, up to 50 core memories — plus the last 10 analyses formatted as history, so the model can reason about trends and avoid repetitive coaching messages.
  - On a state transition to `distracted`, only the *first* frame among parallel in-flight analyses acts (`lastNotifiedState` guard): saves a `focus_sessions` row + a tagged `memories` row locally (SQLite), syncs both to the backend as a memory (`category: system`, tags `["focus","distracted","app:<name>"]`), fires the red glow via `OverlayService`, and (if `notificationsEnabled`) sends a coaching notification through the floating control bar. It then starts a configurable cooldown (`cooldownIntervalSeconds`, default 10 min) during which only a context switch can force re-analysis.
  - On transition back to `focused` *from* `distracted`, it fires the green "refocus" glow and an optional congratulatory notification; transitioning to `focused` from unknown state is silent (no glow/notification spam on first analysis).
  - Errors get exponential backoff (5s→…→5min cap) to avoid hammering Gemini on persistent failures (e.g. safety-filter rejections).
  - All local + backend: capture and classification are local (Gemini call via backend proxy), session storage is local SQLite mirrored to UserDefaults cache, and each event is also synced to the backend `memories` collection so it's visible/searchable cross-surface.
- **Windows status:** **Absent.** Grepped `src/` for `distracted` and `glow` — no matches outside unrelated graph-coloring code (`BrainGraph.tsx`, `nodeColor.ts`) and an onboarding step (`AutoCreatedTasksStep.tsx`) that uses "distracted" as unrelated copy. There is no per-screenshot focused/distracted classification, no cooldown/session model, and no Focus-specific settings tab. Windows' closest relative is `src/main/usage/foregroundMonitor.ts` + `category.ts` — pure foreground-app time tracking with a static app→category table (see "Spotted outside my scope"), which has no LLM judgment and does not know whether the user is *actually* working vs. idly tabbed to a work app.
- **Value / notes:** High — this is one of the two headline proactive features (paired with Insight) and the only one that gives real-time behavioral nudges. Needs Windows equivalents for: active-window screenshot capture per-frame (Rewind capture already exists and could feed this), a glow overlay (see next section), and a notification delivery path (`toastWindow.ts` already exists and could carry Focus alerts). macOS gotcha: `OverlayService` positions 4 separate borderless click-through `NSWindow`s around the target window's Accessibility-API frame — a Windows port would need an equivalent (WS_EX_LAYERED + WS_EX_TRANSPARENT layered windows, or a single click-through overlay window positioned via `GetWindowRect`/UI Automation).

## Focus glow overlay

- **What it is:** A soft animated colored border (green for "back on focus", red for "distracted") drawn around the edges of the user's currently active window for ~3.5 seconds, without stealing focus or blocking clicks into the window underneath.
- **Where (Mac):** `UI/GlowEdgeWindow.swift` (4 borderless click-through `NSWindow`s, one per edge, positioned via Accessibility API + CGWindowList fallback), `UI/GlowBorderView.swift`/`GlowEdgeView` (SwiftUI `MeshGradient` animation on macOS 15+, `AngularGradient` fallback), `Services/OverlayService.swift` (orchestrates show/dismiss, respects `AssistantSettings.glowOverlayEnabled`), `UI/GlowDemoWindow.swift` (dev preview tool).
- **How it works:** Fully local/client-side — no network calls. `OverlayService.showGlowAroundActiveWindow` resolves the frontmost app's focused-window frame (Accessibility API primary, `CGWindowList` largest-window fallback), then creates 4 `GlowEdgeWindow`s (`.popUpMenu` level, `ignoresMouseEvents = true`, `.canJoinAllSpaces`) sized/positioned to hug each edge of that frame with a slight overlap for seamlessness. Each edge hosts a SwiftUI view with a hue-shifting animated gradient (green hue ~0.38, red hue ~0.0) that fades in, pulses for ~1.5s×3, fades out, and self-dismisses after 3.5s (or immediately if a new glow supersedes it).
- **Windows status:** Absent. No overlay window infrastructure for this exists; Windows does have acrylic/mica `BrowserWindow`s for the insight/meeting toast (`toastWindow.ts`) and a separate top-edge bar/orb (per the Windows baseline), but nothing that outlines the *active window itself*.
- **Value / notes:** High — this is the single most visible/distinctive piece of the Focus feature (it's what makes the coaching feel "ambient" rather than a popup). Platform gotcha: this needs Win32 layered, click-through, always-on-top windows positioned to hug an arbitrary foreign window's bounds — more involved on Windows than macOS's Accessibility-API window-frame lookup (would likely need UI Automation `GetWindowRect` + `SetWindowPos`/`SetWinEventHook` to track the target window if it moves during the glow's lifetime, which the Mac version does not even handle since it's a one-shot 3.5s effect).

## Focus session history, stats, and dashboard

- **What it is:** A local history of every focused/distracted transition with computed daily/all-time stats (focused vs. distracted minutes, focus rate %, session count, top distraction apps by time), and a dedicated dashboard page to browse/search/delete sessions.
- **Where (Mac):** `Assistants/Focus/FocusStorage.swift` (`StoredFocusSession`, `FocusDayStats`, SQLite-backed with UserDefaults cache, duration computed by diffing consecutive session timestamps), `MainWindow/Pages/FocusPage.swift` (status banner reflecting live delay/cooldown/analyzed state, stats grid, top-distractions list, searchable session history with delete), `MainWindow/Components/FocusSummaryWidget.swift` (compact Today/Total tab widget reused elsewhere).
- **How it works:** All local — `FocusStorage` loads from SQLite (`ProactiveStorage.shared.getFocusSessions`, last 30 days) on init, caches to UserDefaults for fast cold start, and recomputes stats from the newest-first session list (each session's duration = time until the next one, or now for the most recent). `FocusPage` is a live view: a 1s countdown timer shows the current analysis-delay or cooldown remaining, driven directly off `FocusStorage`'s published `delayEndTime`/`cooldownEndTime`.
- **Windows status:** Absent — no equivalent storage, stats computation, or page exists (there is no Focus data model of any kind on Windows).
- **Value / notes:** Medium — depends entirely on the Focus Assistant existing first; not separable from it.

## Insight Assistant (Proactive Insights)

- **What it is:** A periodic (default every 10 min on Mac / user-configurable 5–60 min on Windows) background pass that reviews recent screen activity and surfaces at most one high-value, non-obvious tip — e.g. "you stashed changes 2h ago, remember `git stash pop`" — as a notification, explicitly avoiding generic/obvious advice.
- **Where (Mac):** `Assistants/Insight/InsightAssistant.swift` (actor, ~1030 lines), `InsightModels.swift`, `InsightStorage.swift`, `InsightAssistantSettings.swift`, `UI/InsightPromptEditorWindow.swift`, `UI/InsightTestRunnerWindow.swift`.
- **Where (Windows):** `src/renderer/src/lib/insightEngine.ts` (orchestration), `insightActivity.ts` (OCR summarization), `insightPrompt.ts` (prompt + schema + response parsing), `insightGate.ts` (confidence + dedup gate), `src/main/insight/{state,notification,toastWindow}.ts`, `src/main/ipc/insight.ts`, `src/main/ipc/db.ts` (`insights` table), `InsightToast.tsx`.
- **How it works (Mac — two-phase, tool-calling, vision-grounded):**
  1. Builds an activity summary directly from the local `screenshots` SQLite table (`GROUP BY appName, windowTitle` over the lookback window, top 30 rows with counts/est. duration) — not just the most recent frame's OCR.
  2. **Phase 1 (text-only investigation loop, up to 7 iterations):** gives Gemini `execute_sql` (arbitrary read-only SQL against the `screenshots` table incl. `ocrText`), `request_screenshot`, and `no_advice` as tools with `thinkingBudget: 1024`; the model actively queries OCR text across the top apps (explicitly told to check 3-5 apps, not just the dominant one, and skip apps with <10 screenshots) before deciding whether anything is interesting enough to look at visually.
  3. **Phase 2 (single vision call):** loads the actual screenshot image the model asked for (compressed to ≤1280px/JPEG q0.4), sends it with the phase-1 findings, and requires the model to cross-reference via more SQL (e.g. "did the user already fix this in a later screenshot?") before calling `provide_advice` — so it won't surface stale advice about something already resolved.
  4. Dedup is both prompt-based (last 30 of up to 50 previous insights, persisted across restarts by reloading tagged `memories` from SQLite) and confidence-gated (`minConfidence`, default 0.85, user-configurable).
  5. On success: saves locally, syncs to backend as a `memories` entry (category `interesting`, tags `["tips","<category>"]`, with `confidence`/`reasoning`/`contextSummary`/`currentActivity`/`headline` fields), updates `InsightStorage` for the dashboard, and sends a notification via the floating control bar (headline as the short body).
  6. Prompt context also includes the AI user profile and the user's detected language (adds a language instruction if not English).
- **How it works (Windows — single-shot, text-only):**
  1. `runInsightOnce()` fires on a `setTimeout` loop (interval from settings, default 15 min), gated on `insight.enabled` and Rewind capture being on.
  2. Pulls Rewind frames for the last 60 min (`window.omi.rewindFrames`), filters private windows (`isPrivateWindow`) and denylisted contexts (`isDeniedContext` — substring match against a fixed + user-editable keyword list), redacts PII patterns (emails, card numbers, SSNs, JWTs, hex/API-key-shaped tokens) from `ocrText`/`windowTitle`, then groups into per-app/window OCR blocks capped at 12,000 chars (`summarizeActivity`) — first block always included even if it alone exceeds the budget, later blocks dropped once the cap is hit.
  3. One Gemini call (`gemini-2.5-flash` by default) with that text summary + last 30 headlines for dedup, structured-output schema requiring `has_insight`/`headline`/`advice`/`reasoning`/`category`/`source_app`/`confidence`. **No image is ever sent** — `insightPrompt.ts`'s `INSIGHT_RESPONSE_SCHEMA` and the `generate()` call only pass `{text: ...}` parts.
  4. `selectInsight` (in `insightGate.ts`) applies a hardcoded confidence threshold (0.85, matching Mac's default but not user-adjustable) and headline-based dedup against the last 30.
  5. On success: inserts into local SQLite `insights` table (dedup history only — **never synced to the backend**), shows via `insight:show` → acrylic toast (`showInsightToast`, 8s auto-dismiss, pauses on hover) or a native Windows notification depending on `notificationStyle`.
- **Windows status:** **Present, but materially weaker.** Concretely missing vs. Mac: (a) no vision/screenshot analysis at all — pure OCR text, so anything not captured well by OCR (visual state, colors, dialog icons, non-text UI) is invisible to it; (b) no tool-calling SQL investigation loop — one shot over a flat 60-min/12k-char text blob vs. targeted multi-app OCR queries; (c) no cross-reference-before-advising step, so Windows can surface advice about something already resolved; (d) confidence threshold and prompt are hardcoded, not user-tunable; (e) no AI user profile / goals / tasks context in the prompt; (f) insights never reach the backend, so they don't show up in chat/RAG context, aren't visible on mobile/web, and don't persist as a permanent memory — only a local dedup cache; (g) no dashboard/history page (see next section).
- **Value / notes:** Medium — the feature exists and is usable today, but the quality ceiling is much lower without vision + investigation, and the lack of backend sync means Windows insights are a dead end (never contribute to the user's broader memory graph). This is the single highest-leverage item to close since the scaffolding (Gemini client, Rewind frames, toast delivery) already exists on Windows — it's an upgrade to `insightEngine.ts`'s extraction quality and a backend-sync addition, not a from-scratch build.

## Insight history / browse UI

- **What it is:** A dedicated page listing all past insights with unread count, category filter tabs (productivity/communication/learning/other, health kept for legacy), search, mark-as-read / mark-all-read, dismiss (hide but keep), delete (permanent, syncs to backend), and a detail sheet showing the full advice + reasoning + context (source app, current activity, context summary) + confidence + relative timestamp.
- **Where (Mac):** `MainWindow/Pages/InsightPage.swift` (`InsightViewModel` + `InsightCard` + detail sheet), backed by `InsightStorage.swift` which syncs from the backend `memories` API filtered by the `tips` tag (`includeDismissed: true`) so read/dismissed state persists cross-device.
- **Windows status:** Absent. Windows persists insights to a local `insights` SQLite table (`src/main/ipc/db.ts`), but the only reader of `recentInsights()` is `insightEngine.ts` itself (for the dedup headline list) — nothing in the renderer ever lists, searches, or lets the user revisit past insights. Once a toast auto-dismisses after 8s, that insight is effectively gone from the user's view.
- **Value / notes:** Medium — a natural pairing with closing the backend-sync gap above (once insights are backend memories, a browse page is mostly a filtered memories view, similar to how the Mac page is really a `memories` view with `tips` tag filtering).

## Spotted outside my scope

- `src/main/usage/foregroundMonitor.ts` + `category.ts` + `usageAccumulator.ts` + `usageDay.ts` — Windows-only static foreground-app time tracker with an app→category table; related in spirit to "focus" but it's dumb time-accounting, not LLM-judged distraction detection — could be a building block for a Windows Focus assistant but is a distinct feature, likely owned by whichever area covers usage/analytics.
- `MainWindow/Components/DailyScoreWidget.swift` / `ScoreWidget` — this is a **task-completion** score ("X of Y tasks completed this week"), not a focus/attention score; despite being near `FocusSummaryWidget.swift` on the Mac dashboard, it belongs to the Task/Goal assistant area.
- `Assistants/MemoryExtraction/MemoryAssistant.swift` and `Assistants/TaskExtraction/*` / `Assistants/Goals/*` — other proactive assistants registered in the same `ProactiveAssistantsPlugin`/`AssistantCoordinator` pipeline; explicitly out of scope per the brief (owned by other agents), but worth noting they share the exact coordinator/backpressure/context-switch infra documented above, so a Windows port of that framework would benefit all of them at once, not just Focus/Insight.
- `Services/AIUserProfileService.swift` — generates the "who this user is" profile text injected into Focus/Insight/other prompts; likely a Memory-area concern, flagged here only because its absence is part of why Windows' Insight prompt is thinner.
- `src/main/screenSynth/state.ts` (referenced in `RewindTab.tsx` as "Synthesize now") — a separate memory-extraction-from-screenshots pass, closer to the Mac `MemoryAssistant`/`TaskAssistant` territory than Focus/Insight.
