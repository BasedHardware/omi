Agent readiness review

Omi is already agent-readable. The missing part is agent-governable.

Hermes can plausibly connect to Omi today through MCP or HTTP-backed tools to search memories and conversations. The stronger product experience is a first-class agent control plane: guided setup, scoped permissions, observable writes, and APIs for the same Memory Assistant settings users can edit in the desktop app.

8+ MCP tools for profile, memory, and conversation access.

Hermes Can add HTTP or stdio MCP servers and has external memory provider support.

What Works Today

Omi already exposes the core primitives an outside agent needs: authenticated memory retrieval, semantic search, memory mutation, conversation search, and a desktop bridge with local context.

Cloud MCP

Hermes can connect as an MCP client

The backend hosts a Streamable HTTP MCP endpoint at /v1/mcp/sse. It authenticates with an omi_mcp_ API key and advertises tools such as get_user_profile, get_memories, search_memories, create_memory, and search_conversations. 

hermes mcp add omi \
  --url https://api.omi.me/v1/mcp/sse \
  --auth header
Local MCP

The desktop app has a local agent bridge

The bundled desktop agent exposes local tools over stdio and HTTP MCP. It can query the local omi.db, semantic-search screen history, produce daily recaps, manage tasks, and relay backend RAG calls through Swift.

  • execute_sql for structured local data.
  • semantic_search for screen history.
  • get_memories and search_memories through backend tools.
Settings exist

Memory behavior is already configurable

The macOS Memory Assistant has user-facing controls for enablement, extraction interval, confidence threshold, custom extraction prompt, notifications, and excluded apps. Settings sync to the backend and server values override local values when present.

How Hermes Would Use It

The nearest-term Hermes story is configuration, not a deep integration rewrite. Hermes already supports MCP server management, header auth, and a built-in memory command.

1. Create an Omi MCP key

Omi exposes key management at /v1/mcp/keys; the user-facing docs point users to Settings > Developer > MCP. Hermes needs the generated omi_mcp_... token as an Authorization header.

2. Add the hosted MCP server

Hermes can add URL-based MCP servers. After adding Omi, hermes mcp test omi should verify discovery and tool calling.

3. Search account memories

For facts and preferences, Hermes should call get_user_profile first, then search_memories or get_memories. For incidents and events, it should use search_conversations.

4. Use desktop context when local app is running

For screen history, local SQLite, local tasks, and daily recaps, Hermes should connect to the desktop bridge MCP server instead of the cloud-only MCP endpoint.

Delta To Great

The product gap is not "can an agent read Omi?" It can. The gap is "can a user safely delegate memory policy to an agent and understand what happened afterward?"

Missing

Hermes one-click setup

Add a Hermes preset or copyable command in Omi Developer settings that creates or selects an MCP key, registers https://api.omi.me/v1/mcp/sse, and runs a test query. Today the pieces exist, but the user must stitch them together.

Missing

Agent-accessible memory settings

The Memory Assistant settings exist in Swift and sync through backend assistant settings, but MCP does not expose policy tools like get_memory_policy, update_memory_policy, add_excluded_app, or set_memory_prompt.

Needs hardening

Scopes and approval semantics

Omi's hosted MCP declares read/write scopes and destructive hints. A polished experience should split keys into read-only, memory-writer, settings-writer, and destructive-delete scopes, with visible approval history for every agent write.

Needs hardening

Memory provenance and review queue

Agents can create, edit, and delete memories today. The better default is staged writes: proposed memory changes show source agent, prompt, reason, source evidence, before/after diff, and allow/deny from Omi.

Recommended next product shape

Agent Control Plane

Add a single "Agents" settings area that lists connected agents, granted scopes, recent reads/writes, memory policy, and test buttons. Under the hood this can reuse the existing MCP key store, hosted MCP endpoint, assistant settings sync, and desktop bridge. The new work is mostly product glue, settings tools, audit trails, and safer defaults.

Readiness Scorecard

A practical read on how close Omi is today for a local agent like Hermes.

Strong

Memory search and retrieval

Hosted MCP, Python stdio MCP, backend REST tools, and desktop backend RAG tools all support memory retrieval or search.

Strong

Conversation recall

Omi distinguishes static facts from event recall. The tool descriptions explicitly steer agents toward memories for facts and conversations for events.

Medium

Behavior customization

Users can customize memory extraction in the desktop UI today, but outside agents cannot yet do it through a safe, documented Omi tool surface.

Medium

Local context access

The desktop bridge is powerful, but discovery and external client attachment are more implicit than the hosted MCP path.

Low

Audit and governance

The code has scopes and destructive hints, but not yet a user-facing ledger of agent reads, writes, policy changes, and rollback.

Low

Hermes-specific polish

Hermes is locally installed and supports MCP add/test commands, but Omi does not yet ship a Hermes preset, setup copy, or compatibility test.
